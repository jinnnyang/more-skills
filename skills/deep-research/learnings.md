# Learnings (深度研究技能学习经验沉淀)

为智能体提供跨会话的执行经验与已知陷阱规避。

## ⚠️ 已知环境与系统问题 (Known Environment Issues)
- 外部网站爬取可能存在高延迟或超时。
- 读写本地文件时，如果编码不一致可能会触发编码冲突（强制使用 UTF-8）。
- 本地可能未安装 Git 命令行客户端（作为 warning 跳过，不应阻断运行）。

## 🎯 成功模式 (Success Patterns)
- **工作区作为长期记忆**：强制主智能体与子智能体将线索、发现、猜测等记录在 `memories/`、`findings/`、`clues/`、`hypotheses/` 扁平共享文件夹中，并周期性同步与主动抛弃旧上下文（如网页抓取原始文本），这能有效防止超大型项目中的上下文溢出。
- 优先分发给 `research` 子代理进行网络搜索，以最大化释放主智能体（Lead Agent）的上下文空间。
- 在 P5 阶段严格执行按大纲章节顺序循环生成与流式追加写入（Loop-Append），以彻底规避单次生成时大文本输出被截断的限制。
- 每个事实论点末尾必须附带全局编号引用（如 `[n]`），并在附录中生成完整对应的引文。

## ❌ 失败与反模式教训 (Failures & Anti-patterns)

为了防范大模型在执行深度调研时的失控或质量退化，必须严格避开以下 14 条反模式：

1. **❌ 一次性完整撰写大文本草稿**：严禁一次性输出整篇数千字的报告。这会导致模型在输出中途被截断，并使 Python/Markdown 语法解析崩溃。必须严格执行按大纲章节流式追加拼接（Loop-Append）。
2. **❌ Lead Agent 直接阅读原始搜索结果**：主代理必须仅阅读子代理提炼过的 Findings 与 Notes。严禁将原始搜索引擎返回的大文本网页抓取结果塞入 Lead Agent 上下文，防止其工作记忆过载并导致生成质量退化。
3. **❌ 虚构或凭空捏造 URL 链接**：参考文献列表与多媒体中的 URL 链接必须绝对来自子任务的真实网络抓取结果，严禁智能体自行脑补或猜测 URL。
4. **❌ 复活已丢弃的文献源 (Dropped Sources)**：在 P3 阶段因质量不符（如时效性极差、来源不权威等）被剔除并放入 `Dropped` 列表的信源，严禁在后续 P5 大纲和草稿编写中再次引用。
5. **❌ 时间敏感的论断缺少时效戳记**：在陈述政策、市场份额、技术指标等高时效性事实时，严禁忽略数据时间。必须引述对应文献的发布时间，并与 P0 设定的 `AS_OF` 截止时间进行交叉评估。
6. **❌ 循环验证 (Circular Verification)**：严禁调用用户的私有数据账户（如域名的 registrar 账户）去“发现”用户本来就已经知道的自己公司的基础数据。当调研用户自身企业时，必须使用外部公共视角（WHOIS、公开报道等）。
7. **❌ 忽视专有/排他性信息优势 (Exclusive Sources)**：当用户为第三方竞争调研主动提供 Crunchbase Pro 等专有订阅源或私有数据 API 时，严禁忽视它们。应当将其接受为 `exclusive-user-provided` 并深度使用。如果数据无公共来源验证，在报告中需标记为 `[unverified]`。
8. **❌ 略过 P6 阶段的反方立场审查 (Counter-Review)**：反方审查是强制作业（Mandatory），严禁敷衍略过。主代理必须发现并明确记录至少 3 个高价值的对立观点、反向证据或潜在的数据局限性。
9. **❌ 正文引文缺失或对账不一致**：正文中的每一个事实结论必须附带 `[n]` 序号引文标记。引文序号必须与报告尾部的参考文献表和多媒体注册表一一对齐，严禁序号断档或张冠李戴。
10. **❌ Mermaid 代码块渲染报错**：正文中嵌入的 Mermaid 图表，其节点文本必须使用双引号包裹（例如 `A["节点文本"]`），且 ID 严禁包含特殊字符，以防止渲染器崩溃。
11. **❌ 硬编码个人或本地绝对文件路径**：技能文件和生成文档中的所有跳转链接必须使用**标准相对路径**（例如 `[p0_setup.md](references/phases/p0_setup.md)`），严禁使用类似 `file:///C:/Users/jinnn/...` 的硬编码绝对路径。
12. **❌ 缺失多媒体佐证的详尽 Alt 可读性描述**：报告正文和附录中引用的多媒体图片，其 Markdown Alt 括号内必须包含详尽的“图表类型、核心指标、数据趋势”解释，以便下游大模型（非多模态或断网状态）能够正确阅读并理解该图表内容。
13. **❌ 企业模式中遗漏 L1/L2/L3 严格的卡点质检**：在企业模式下，严禁漏掉任何一级质量核验（数据级、分析级、文档级），且 SWOT、竞争壁垒打分及风险矩阵的每一个条目必须直接关联并追溯到数据事实与引文编号。
14. **❌ 网络超时或 Git CLI 缺失时抛错中断**：在网络偶发性超时（如 EuropePMC 连接超时）或本地没有 Git 命令环境时，严禁作为 Critical 报错抛出并中断主运行。必须将其记为 Warning 日志自动跳过，确保主干任务能顺利走完。
15. **❌ 跨项目关联时发生上下文穿舱 (Context Overflow)**：在执行延续性研究或读取其他项目的 Findings/Report 时，严禁毫无节制地加载旧项目下所有的海量原始 notes 或抓取全文。只能按需指定读取高密度的最终 `report.md`，防范历史负担拖垮当前轮次的生成质量。
