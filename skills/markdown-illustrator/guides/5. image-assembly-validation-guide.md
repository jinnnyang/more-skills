# 🔩 Markdown 插图装配与格式校验指南 (Image Assembly & Validation Guide)

本指南规范了 Markdown 插图工作流的 **步骤 4 (插图装配/Assembly)** 和 **步骤 5 (格式校验/Validation)** 的执行细节。

经过前面步骤的分流处理，各子 Guide 已在隔离上下文中生成了图片，并以统一格式记录在 `*.cand.md` 清单中。本模块的职责是：
1. 读取并解析 `*.cand.md`。
2. 采用**自底向上装配算法**将图片引用插入到原始 Markdown 对应的行号。
3. 对装配后的文档进行格式编译（如 Mermaid 转 PNG）与完整性校验。

---

## 📂 一、 数据契约：`*.cand.md` 结构规范

为了实现上下文隔离时的状态聚合，所有子 Guide 在生成完图片后，必须将产出以以下**单行标准格式**追加到 `[原文章名].cand.md` 中：

```markdown
<建议插入行号>: ![<Alt 文本>](<图片路径或网络URL> "<Title 标题>")
```

### 规范示例
```markdown
23: ![新华社记者摄：2024年3月11日下午，十四届全国人大二次会议在北京人民大会堂闭幕。](https://www.news.cn/politics/2024lh/images/closing.jpg "两会闭幕会现场-新华社")
48: ![瑞士风双栏对比图：较高层级与一般层级党员干部学习方法对比](images/leaders-methods-card.svg "党员领导干部分类学习贯彻指南")
105: ![Mermaid流程图：2024年全国两会学习贯彻逐级向下传导与全覆盖机制](images/overall-process-flowchart.mmd "全景穿透式贯彻流程图")
```

> [!IMPORTANT]
> - 本地生成的所有 SVG、PNG、MMD 文件必须存放在主文档同级目录下的 `images/` 文件夹中。
> - 网络检索的纪实图片采用远程 URL 热链，无需下载。

## 🛠️ 二、 步骤 4 — 智能装配 (Assembly)

在装配阶段，Agent 必须读取原始 `.md` 文件和对应的 `*.cand.md` 清单，执行以下装配流程：

### 1. 行号漂移与自底向上插入算法 (Bottom-up Insertion)
如果按照行号从小到大的顺序在原文中插入图片引用，每一次插入都会导致后续所有行的行号增加，从而引发**行号漂移 (Line Drift)** 错误。

**执行规则**：
- **步骤 A**：读取 `*.cand.md` 中的所有行，解析出行号和图片引用代码。
- **步骤 B**：将所有解析后的条目**按行号进行降序排序 (Descending Order)**（从大到小）。
- **步骤 C**：从行号最大的条目开始，依次将图片引用代码插入到原始 `.md` 文件的对应行（在指定行号的**下一行**新建一行插入）。
- **示例**：若有 23 行和 105 行两个插入点，先在第 105 行后插入，再在第 23 行后插入。这样，第 23 行的绝对行号在第 105 行插入后不会受到任何影响。

## 🧪 三、 步骤 5 — 格式编译与完整性校验 (Validation)

装配完成后，必须对文档和物理资产进行完整性自检，确保渲染无误。

### 1. Mermaid 编译 (Mermaid to PNG Compilation)
在 Markdown 原文中直接内嵌图片引用 `![alt](images/xxx.mmd "title")` 时，许多 Markdown 阅读器（如微信公众号后台、部分网页端）无法直接渲染 `.mmd` 源码。

**编译规则**：
- 扫描装配后的 Markdown 链接，若发现后缀为 `.mmd` 的图片引用：
  - 使用本地脚本工具 `scripts/mermaid-tool.ts` 对其进行校验并编译为高分辨率 `.png`。
  - **编译命令示例**：
    ```bash
    npx tsx scripts/mermaid-tool.ts -i images/xxx.mmd -o images/xxx.png -t zinc-light
    ```
  - 编译成功后，将主文档中的图片引用后缀由 `.mmd` **更新为 `.png`**，即：
    - 修改前：`![alt](images/xxx.mmd "title")`
    - 修改后：`![alt](images/xxx.png "title")`
  - 保留原始 `.mmd` 文件在 `images/` 目录下，便于后续修改，但主文档中只引用 `.png`。

### 2. 物理资产存在性校验 (Asset Integrity Check)
- 检查主文档中引用的所有本地图片文件（如 `images/xxx.svg`、`images/xxx.png`）是否确实存在于磁盘的 `images/` 文件夹下。
- 若有缺失，必须立即报告具体的缺失文件名，并标记该配图节点为 `Failed`。

### 3. 双轨制 Alt 文本合规性校验 (Alt Metadata Validation)
依据《媒体资产检索与真实证据配图指南》的**视觉双轨制**，对插图的 Alt 文本进行合规性抽查：
- **规则 A (纪实证据图)**：其 Alt 文本必须包含事实元数据（例如：图源、时间、地点、具体新闻事件描述）。严禁使用过于笼统或主观的描述。
- **规则 B (设计卡片/Mermaid/AI插画)**：其 Alt 文本必须清晰描述卡片的核心文字或图表的逻辑结构，以确保可读性与无障碍支持 (a11y)。
- **禁止项**：Alt 文本严禁为空或仅写为 `"图片"`、`"插图"`。

### 4. 远程热链连通性校验 (Remote Hotlink Integrity Check)
- 扫描装配后的 Markdown 链接，针对任何以 `http://` 或 `https://` 开头的图片 URL：
  - Agent 必须使用其 HTTP 或网络工具发起一个完整的 **HTTP GET 请求**。
  - **校验条件**：
    1.  网络请求状态码必须为 `200 OK`。
    2.  响应头的 `Content-Type` 必须以 `image/` 开头（例如 `image/jpeg`、`image/png`、`image/webp` 等），确保该 URL 确实指向一个有效的图片资产，防止发生远程图片失效或被防盗链阻断。
  - 若请求超时、返回非 200 状态码、或 `Content-Type` 不匹配，必须将该配图节点标记为 `Failed` 并报告具体失效 URL，阻断整个校验流程。

## 🏁 四、 输出与校验报告 (Verification Report)

在步骤 5 执行完毕后，Agent 应当在控制台或对话中输出一份简要的装配校验报告，格式如下：

```markdown
## 🏁 插图装配与校验报告

- **目标文档**：`[文章名].md`
- **候选清单**：`[文章名].cand.md` (共 N 张候选图)
- **装配状态**：
  - [x] 位置 1 (行号 X)：装配成功 (AI 生成, PNG)
  - [x] 位置 2 (行号 Y)：装配并编译成功 (Mermaid .mmd -> .png)
  - [x] 位置 3 (行号 Z)：装配成功 (SVG 卡片)
  - [x] 位置 4 (行号 W)：装配成功 (网络热链, 证据级 A)
- **格式校验**：
  - [x] 所有本地图片物理存在
  - [x] Alt/Title 文本双轨制合规性通过
  - [x] Mermaid 渲染引擎输出无报错
```

通过此报告，工作流即可完美过渡到**第六步**（调用 `1. image-text-evaluation-guide.md` 进行最终的配图后质量评估与前后对比）。
